# S3 - the operation entry point

`S3` is the public entry point for the `aws s3`-style operations
(`cp` / `ls` / `mv` / `rm` / `mb` / `rb` / `presign` / `sync` / `website`). It is
a **stateless orchestrator**: it carries optional defaults for building clients
and transfers but holds no connection, so it needs no cleanup. Each method's
`Location` arguments are resolved to a `Storage` (an `S3Storage` or a
`LocalStorage`); the building blocks it drives are documented in
[`transfer.md`](./transfer.md), [`sync.md`](./sync.md), and
[`deleter.md`](./deleter.md).

Two methods - `client()` and `resolve()` - are the **customization seams**:
subclass `S3` and override them to change how clients are built or how URLs are
interpreted. They are also why constructing an `S3` is cheap and SDK-free - the
boto3 client is built only when `client()` is actually called
([`imports.md`](./imports.md)).

## 1. API

```python
import boto3
from boto3_s3 import S3

S3().cp("local.txt", "s3://bucket/key")                       # zero config
S3(endpoint_url="https://minio.local:9000").ls("s3://bucket/")  # MinIO
S3(session=boto3.Session(profile_name="prod")).sync("s3://a", "s3://b")
```

| Member | Description |
|---|---|
| `S3(session=None, *, endpoint_url=None, config=None, transfer_config=None)` | Holds the defaults: `session` (a `boto3.Session`; default = boto3's default session) plus `endpoint_url` / `config` (a `botocore.config.Config`) for building clients, and a default `TransferConfig` for `cp` / `mv` / `sync`. All optional; `S3()` is the zero-config form. Construction is SDK-free (no boto3 import, no connection). |
| `client() -> S3Client` | The client factory (override seam). Builds a **fresh** boto3 S3 client from the instance's `session` / `endpoint_url` / `config`, **owned by the caller** - `S3` keeps no connection, hence no `close()`. Override to change credentials, memoize/reuse a client, or return a test double. |
| `resolve(loc) -> Storage` | The URL-interpretation seam (used by `cp` / `mv` / `sync`). A `Storage` instance is returned verbatim; an `"s3://..."` string becomes an `S3Storage` carrying `client()`; anything else is a `LocalStorage` (aws-cli's rule: only `s3://` is special). Override to add schemes such as `http://`, deferring the rest to `super().resolve(loc)`. Its inverse is `Storage.as_text()` - the canonical aws-cli path-shape token that `str(storage)` returns and the transfer engine's `naming` layer consumes. |
| `aws_config() -> AwsConfig` | Reads this instance's AWS config file (`~/.aws/config`) - an **explicit opt-in** building block; the operations never read it themselves (no hidden ambient-config dependence). Returns an `AwsConfig` reader for pulling out profile settings the app did not set (e.g. `[s3]` transfer tuning). Memoized per instance. See section 5. |
| `cp` / `ls` / `mv` / `rm` / `mb` / `rb` / `presign` / `sync` / `website` | The operations, mirroring `aws s3 <sub>`. Where a `transfer_config` argument is accepted it defaults to the instance's `transfer_config` and is overridden per call. |

The instance is **safe to share across threads**: it carries immutable defaults
plus one benign, idempotent cache (the `aws_config()` reader, parsed once per
instance - concurrent first calls just recompute the same reader). Concurrency
beyond that is bounded by boto3, not by `S3`: `client()` builds a fresh client
each call, and building clients concurrently from a shared session is boto3's
own not-thread-safe path (duplicate credential resolution; the shared
event-emitter copy). For heavy parallelism reuse one prebuilt client via
`S3Storage(url, client=...)`, or override `client()` to memoize.

## 2. Resolution and client inheritance

A bare string argument inherits this `S3`'s client defaults; an explicitly
constructed `Storage` is used as-is.

| Argument | Resolves to |
|---|---|
| `"s3://bucket/key"` (`cp` / `mv` / `sync`) | `S3Storage` carrying `client()` |
| `"local/path"`, `"./x"` (`cp` / `mv` / `sync`) | `LocalStorage` |
| `"bucket/key"` or `"s3://..."` (the S3-only ops: `ls` / `rm` / `mb` / `rb` / `presign` / `website`) | `S3Storage` carrying `client()` - these ops are **lenient** about the `s3://` prefix |
| an `S3Storage(url, client=...)` you pass in | used verbatim (its own client, **not** this `S3`'s) |
| an `IOStorage(stream)` / `StdioStorage()` you pass in (`cp`) | used verbatim - a stream side, transferred via its `open()` (the other side must be S3) |

Pass an explicit `S3Storage(url, client=...)` when a single operation needs more
than one client - e.g. **S3-to-S3 across accounts** (a different client per
side), or a per-location endpoint. The instance default applies only to bare
strings.

The strict `s3://`-vs-local split lives in `resolve`; the **CLI layer** enforces
the stricter per-subcommand `s3://`-required rules (`aws s3` parity) on top - the
library stays the more permissive building block (see
[`overview.md`](./overview.md) section 3).

## 3. Module-level convenience functions

`boto3_s3.cp` / `ls` / `mv` / `rm` / `mb` / `rb` / `presign` / `sync` /
`website` are thin wrappers over a default `S3()`, for the common zero-config
case (`boto3_s3.sync(...)` instead of `S3().sync(...)`). Each keeps its method's
exact signature for type checkers and IDEs (a `ParamSpec` delegator, with the
method as the single source of truth). They build a fresh `S3()` per call - there
is no shared singleton; for configured behavior use `S3(session=...).sync(...)`.

## 4. Customization by subclassing

```python
class MyS3(S3):
    def client(self):                      # custom credentials / a mock in tests
        return my_session.client("s3", config=my_config)

    def resolve(self, loc):                # add an http:// backend
        if isinstance(loc, str) and loc.startswith("http://"):
            return HttpStorage(loc)
        return super().resolve(loc)
```

Overriding `client()` to memoize a client makes that subclass own a connection;
it then takes on the cleanup (a `with` / `close()` of its own). The base `S3`
deliberately keeps none, so it stays thread-safe and lifecycle-free.

The S3-only operations (`ls` / `rm` / `mb` / `rb` / `presign` / `website`) accept
only an `S3Storage` (a subclass customizing S3 behavior included), never an
arbitrary custom `Storage` - each needs an S3 bucket and client. A **stream**
Storage (`IOStorage` / `StdioStorage`) is transferable today: `cp` reads / writes
it through its `open()` (one S3 side plus the stream). Making a `Storage`-direct
custom *container* (e.g. an HTTP backend) transferable through `open()` is a
separate, still-pending seam (one S3 side; `cp` / `mv` / `sync` only) - see
`storage.py`'s module docstring.

## 5. Reading the AWS config file

`s3.aws_config()` exposes the AWS config file (`AWS_CONFIG_FILE`, default
`~/.aws/config`) as an `AwsConfig` reader. It is a **general** reader - not
limited to `[s3]` - that pulls any value the app did not set itself out of the
profile (or another section). The parse is delegated to botocore
(`Session.full_config` / `get_scoped_config`), so the file location, the nested
`s3 =` subsection syntax, and active-profile resolution match `aws` / `boto3`
exactly; only `~/.aws/config` is read, never the credentials file.

```python
cfg = s3.aws_config()
cfg.get_size("s3.multipart_chunksize", 8 * 1024**2)   # active profile, "16MB" -> bytes
cfg.get_str("region", None)
cfg.profile("prod").get_int("s3.max_concurrent_requests", 10)
cfg.services("my-svc").get_str("s3.endpoint_url", None)  # [services my-svc]
cfg.sso_session("my-sso").get_str("sso_region", None)  # [sso-session my-sso]
cfg.plugins().get_str("cli_legacy_plugin_path", None)
```

Mirroring `aws configure get`, the **profile is the default context** (the bare
getters read the session's active profile) and `services` / `sso_session` /
`plugins` are explicit selectors, each returning a `ConfigSection`. The two
deliberate additions over `aws`'s loader: **typed getters** (`get_str` /
`get_int` / `get_size` / `get_bool` / `get_rate`; sizes/rates are 1024-based to
match `aws s3`) and **caller-supplied defaults**. A missing key or absent named
section returns the default; a present-but-unconvertible value, or a key naming
a whole subsection, raises `ConfigurationError`; a set-but-missing active profile
surfaces botocore's `ProfileNotFound` as a `ConfigurationError`.

This is an **opt-in building block**: `cp` / `sync` / ... never consult it on
their own. `AwsConfig` lives in `boto3_s3.awsconfig` and is imported by submodule
path (`from boto3_s3.awsconfig import AwsConfig`), not the lazy root re-export;
importing it stays SDK-free. Matching `aws s3`'s `[s3]` semantics on top of these
values - the defaults table, validation, the classic/CRT engine decision - is the
**CLI distribution's** responsibility (`boto3_s3_cli.runtimeconfig`), not the
library's.
